💡 La Idea Central: En lugar de intentar enviar el evento a RabbitMQ mientras el cliente espera, lo guardamos en una tabla dentro de PostgreSQL. Si la base de datos funciona, la venta se completa.

10.3 / CONCEPTO
03 · Paso 1

El Guardado Atómico

Todo en una sola transacción

Dentro de un mismo @Transactional, guardamos dos cosas:

Atomicidad Total

1

El Pedido en la tabla t_orders

2

El Evento en la tabla outbox_events (JSON del mensaje) 

@Transactional
public OrderResponse placeOrder(OrderRequest request) {
    // 1. Guardar el pedido
    Order order = orderRepository.save(newOrder);
    
    // 2. Guardar el evento en Outbox (NO enviarlo a RabbitMQ)
    OutboxEvent event = new OutboxEvent(order, "PENDING");
    outboxRepository.save(event);
    
    // ✅ COMMIT - Ambas cosas están seguras
    return toResponse(order);
}

✅ Garantía ACID: Si PostgreSQL dice 'OK', ambas cosas se guardaron. Si falla, ninguna. RabbitMQ ya no participa en la transacción.

10.3 / PASO 2
05 · Paso 3

La Estructura de la Tabla

Qué campos necesita nuestra Bandeja de Salida

Aunque implementaremos la entidad en la próxima clase, así es como se verá la estructura que nos salvará el negocio: 

CREATE TABLE outbox_events (
    id           BIGSERIAL PRIMARY KEY,
    aggregate_id VARCHAR(255) NOT NULL,  -- orderNumber
    event_type   VARCHAR(100) NOT NULL,  -- "OrderPlacedEvent"
    payload      TEXT NOT NULL,          -- JSON del evento
    status       VARCHAR(20) NOT NULL,   -- PENDING / PROCESSED
    created_at   TIMESTAMP NOT NULL,
    processed_at TIMESTAMP
);

Campos Clave

📦

payload: El JSON completo del evento (para enviarlo después)

🏷️

status: PENDING o PROCESSED (para no repetir envíos)

🔗

aggregate_id: El orderNumber (para relacionarlo con la orden)
