Retry Pattern

El Arte de la Insistencia Elegante
¿Es un patrón de diseño o de arquitectura?
Esta es la pregunta que mucha gente no se hace.

¿Qué es realmente el Retry Pattern?

🏛️ Clasificación Técnica
El Retry Pattern se clasifica como un Patrón de Estabilidad o Cloud Design Pattern (Patrón de Diseño en la Nube).
Aunque se implementa en el código (diseño), su impacto y propósito son arquitectónicos.
¿Por qué es un híbrido?
Como Patrón de Diseño: Se parece a un Decorator o Proxy. Envuelves tu llamada original en una lógica que la repite.
Como Patrón Arquitectónico: Es vital para la Resiliencia. Define cómo se comportan tus servicios cuando la red es hostil.

El Concepto Fundamental

Es una estrategia de manejo de errores que asume que ciertas fallas son temporales (transitorias).
"Si falló ahora, quizás en 200 milisegundos funcione"
⚡ La premisa del Retry: No te rindas a la primera. Espera un poco y prueba de nuevo.

¿Por qué es más relevante en Microservicios?

🏢 Monolito
Todo está en una sola memoria.

Si algo falla, suele ser definitivo:

• NullPointerException
• Error de lógica
• Datos inválidos
• División por cero
❌ Sin red = Sin errores transitorios
Los fallos son deterministas y reproducibles.
🌐 Microservicios
Todo está distribuido en red.

El 90% de los errores son de red:

• Un parpadeo en el WiFi
• Balanceador cambiando instancias
• Base de datos bloqueada 5ms
• Timeout por congestión
✅ La red es caótica = Retry es esencial
La mayoría de estos errores desaparecen en milisegundos.
Sin Retry, una arquitectura de microservicios es extremadamente frágil. Un solo fallo en cadena rompe todo.

Las 3 Reglas de Oro

Un buen Retry no es solo un while(true)
1
Count (Cantidad)
¿Cuántas veces insisto?

Ejemplo: 3 veces

Si insistes infinitamente, bloqueas tu propio sistema.
2
Interval (Espera)
¿Cuánto espero entre intentos?

Ejemplo: 2 segundos

Dar tiempo al servicio para recuperarse.
3
Backoff (Retroceso)
Esta es la "inteligencia" del patrón.

Define cómo crece el tiempo de espera entre intentos.

Estrategias de Backoff

La inteligencia del Retry

📊 Fixed Backoff (Fijo)
Espero siempre el mismo tiempo: 2s → 2s → 2s

Problema: Si el servicio está saturado, seguirás bombardeándolo con la misma frecuencia.
🚀 Exponential Backoff (Exponencial) ✨
El tiempo de espera crece exponencialmente: 1s → 2s → 4s → 8s

✅ Best Practice: No saturas al servicio que ya está sufriendo. Le das cada vez más tiempo para recuperarse.
💡 Exponential Backoff es la estrategia recomendada por AWS, Google Cloud y Azure para sistemas distribuidos.

¿Cuándo NO usarlo?

La trampa del Retry mal aplicado
El Retry solo sirve para Errores Transitorios
✅ Úsalo para:
🔌 503 Service Unavailable
⏱️ Timeout
🚫 Connection Refused
📡 Network Error
🔄 429 Too Many Requests
Por qué: Estos errores son temporales. El servicio puede estar reiniciándose, el balanceador rotando instancias, o la red congestionada.
❌ NO lo uses para:
🔐 401 Unauthorized
📝 400 Bad Request
🔍 404 Not Found
🚷 403 Forbidden
💥 500 Internal Server Error
Por qué: Si te equivocaste en la contraseña, por más que reintentes 100 veces, seguirá estando mal. Solo gastarás recursos.

Tipos de Errores en Detalle

Saber distinguirlos es crítico

✅ 503 Service Unavailable
El servicio está temporalmente fuera de línea. Reintentar en 2-5 segundos tiene sentido.
✅ Timeout
La red estaba lenta o el servidor sobrecargado. Puede funcionar en el siguiente intento.
✅ Connection Refused
El servicio está reiniciándose. En unos segundos podría estar disponible.
❌ 401 Unauthorized
Token inválido o expirado. Reintentar no cambiará el resultado. Necesitas un nuevo token.
❌ 400 Bad Request
Los datos están mal formados. El mismo request siempre dará el mismo error.
❌ 404 Not Found
El recurso no existe. Insistir 10 veces no lo creará mágicamente.

Ejemplo Práctico

Escenario Real
Tu sistema hace una petición HTTP a un microservicio de inventario. La red tiene un "parpadeo" de 0.5 segundos y la conexión falla.
❌ Sin Retry
Intento 1: Timeout
Resultado: Error al usuario
Usuario: 😡 "La app no funciona"
Pérdida de negocio
El usuario ve un error por un problema de red de medio segundo. Mala experiencia.
✅ Con Retry (Exponential Backoff)
Intento 1: Timeout (1s)
Espera: 1 segundo
Intento 2: ✅ Éxito
Usuario: 😊 "Todo funcionó"
Resiliencia automática
El usuario ni siquiera sabe que hubo un problema. Sistema robusto.

Entonces:

🎯 Naturaleza: Patrón de Estabilidad / Cloud Design Pattern
💡 Propósito: Manejar errores transitorios en sistemas distribuidos
⚙️ Componentes: Count (cantidad) + Interval (espera) + Backoff (estrategia)
🚀 Best Practice: Exponential Backoff
⚠️ Cuidado: Solo para errores transitorios, no errores de lógica
"El Retry Pattern es el arte de la insistencia elegante.

Es la herramienta que evita que tu usuario vea una pantalla de error
solo porque un cable de red parpadeó durante 0.5 segundos."
